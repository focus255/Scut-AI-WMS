## 1. 系统架构

### 1.1 整体架构

系统采用前后端分离的B/S架构。前端为Vue 3单页应用，通过Axios与后端Spring Boot RESTful API通信。后端采用分层架构：Controller层接收请求并校验参数、Service层处理业务逻辑、Mapper层负责数据持久化。

```
浏览器 / 移动端
     │
     ├── Vue 3 SPA (Element Plus + ECharts)
     │   ├── views/    页面组件
     │   ├── api/      Axios封装
     │   ├── router/   路由守卫
     │   └── stores/   Pinia状态
     │
     ▼ HTTP / JSON
     │
Spring Boot 3.2 (Tomcat)
     │
     ├── interceptor/  JWT鉴权
     ├── controller/   REST接口
     ├── service/      业务逻辑
     ├── mapper/       MyBatis-Plus数据访问
     └── config/       线程池/CORS/安全配置
     │
     ▼ JDBC
     │
MySQL 8.0
```

### 1.2 分层职责

Controller层：接收HTTP请求，参数校验，调用Service，统一返回Result包装。不包含业务逻辑。

Service层：封装所有业务规则。入库服务处理条码生成和库存累加。出库服务处理FIFO拣选和库存扣减。封存服务处理状态互斥。库存报表服务处理四级评级计算。

Mapper层：MyBatis-Plus接口，对应18张数据库表。复杂查询使用LambdaQueryWrapper构建条件。

Interceptor层：JwtInterceptor拦截所有非OPTIONS请求，校验Authorization头中的Bearer令牌。解析成功后注入用户上下文到线程变量。

### 3.2 实体关系

suppliers与materials为一对多关系，一个供应商可供应多种物料。materials与appliances为一对多关系，一种物料在不同供应商下可有不同包装配置。

inbound_orders与inbound_details为一对多关系，一张入库单可包含多条明细。inbound_details与barcodes为一对多关系，一条明细对应多箱。

outbound_orders与outbound_details为一对多关系。outbound_details通过outbound_histories关联到barcodes，形成出库批次与入库条码的追溯链路。

inventories与materials为一对一关系，每种物料一条库存记录。

barcodes与inventory_freezes为一对多关系，一个条码可被多次封存解封。

### 3.3 索引设计

- 物料号在materials、inventories、appliances、barcodes、inbound_details、outbound_details、outbound_histories表中均建立索引
- 入库单号和出库单号在对应主表建立唯一索引
- 条码号在barcodes表建立唯一索引
- 外键字段均建立普通索引加速联表查询
- outbound_histories的created_at建立索引支持FIFO排序和时间范围筛选
- barcodes的type加status加remaining_qty建立复合索引加速拣选查询

## 4. API设计

### 4.1 统一响应格式

所有接口返回统一JSON结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

code为0表示成功。业务错误返回对应错误码和提示信息。前端Axios响应拦截器自动剥离外层包装，业务代码直接获取data字段。

### 4.2 错误码定义

| 错误码 | 语义 |
|:---|:---|
| 0 | 成功 |
| 400 | 请求参数校验失败或业务规则拒绝 |
| 401 | 未登录或令牌失效 |
| 403 | 已登录但权限不足 |
| 404 | 请求资源不存在 |
| 500 | 系统内部异常 |
| 2001 | AI线程池满或任务提交被拒绝 |
| 2002 | AI接口调用超时，已触发本地规则引擎兜底 |
| 3001 | 库存不足 |

## 5. 核心算法设计

### 5.1 FIFO拣选算法

出库拣选是系统的核心算法，确保库存按先入先出原则消耗。算法分为四个阶段，每个阶段内部按FIFO排序。

FIFO排序规则：先按来源入库单的创建时间升序，再按条码创建时间升序，最后按条码ID升序。入库单创建时间较早的条码优先被拣选。同一入库单内的条码也按创建时间严格排序，保证了时间上完全一致的条码不会因为ID顺序而导致拣选结果不确定。

阶段一 整箱选取：遍历FIFO排序后的在库条码，将remainingQty等于箱容量且大于等于需求量的条码整箱全取。整箱全取的条码状态变为待出库，remainingQty保持不变，待确认时清零。

阶段二 部分箱补充：整箱不足时，按FIFO遍历部分箱。扣除量取当前remainingQty和剩余需求量的较小值。扣除量小于remainingQty时条码保持待在库状态仅扣减数量，等于remainingQty时条码归零并标记待出库。

阶段三 拆整箱：以上仍不足时，从FIFO最早未动的整箱中拆出所需数量。为拆出部分生成新的拆分二维码，条码号格式为`WMS|物料|供应商|拆出量|箱容量|拆出量|来源箱序号_S时间戳`。原箱扣减后保持待在库状态。拆分二维码标记为待出库。

拣选阶段不扣减库存，待确认出库时再扣。

### 5.2 库存四级评级算法

评级执行顺序严格按照优先级从高到低：

步骤一：库存量小于等于0时判定为LOW_STOCK。

步骤二：最后出库日期距今不少于90天且库存大于0时判定为DEAD_STOCK。

步骤三：计算补货预警线，公式为日均销量乘以补货提前期加安全库存。当前库存低于预警线时判定为LOW_STOCK。

步骤四：计算DOHF等于当前库存除以日均销量。DOHF大于高储控制天数上限时判定为HIGH。

步骤五：以上均不满足时判定为NORMAL。

五日移动平均平滑：日均消耗计算采用近30天出库总量除以30，无出库历史时返回0。

### 5.3 安全库存建议算法

采用经典库存管理公式，基于需求波动和提前期计算建议安全库存量。

公式：Z乘以σ_daily乘以根号LT

其中Z等于1.65对应95%服务水平。σ_daily为近30天每日出库量的标准差。LT为补货提前期天数。

计算步骤：查询近30天出库流水，按日期分组汇总日消耗量，填充无出库日为0。计算序列标准差。套用公式得出建议值并向上取整。

数据不足5条时回退到日均消耗乘以3天的经验值。

### 5.4 提前期建议算法

统计最近10条已确认入库单中包含该物料的订单，按创建时间升序排列。计算相邻订单间的天数差，取平均值作为建议提前期。数据不足2条时保留手动配置值。

### 5.5 需求趋势检测算法

趋势检测：将12周历史数据按时间分为前6周和后6周。计算前后两段均值。后段均值相比前段增长超过20%判定为上升趋势，下降超过20%判定为下降趋势，否则为平稳。

波动检测：计算12周数据的变异系数CV等于标准差除以均值。分类阈值：CV小于0.3为低波动，0.3至0.7为中波动，大于0.7为高波动。均值为0时CV无效，判定为低波动。

异常检测：单周值与均值的偏差超过2.5倍标准差时标记为异常。

### 5.6 二维码状态机

五种状态及其转移规则：

- 待入库：入库单创建时的初始状态。转移至在库通过确认入库或扫码入库。

- 在库：可供出库的正常状态。转移至待出库通过出库单FIFO拣选。转移至FROZEN通过封存操作。在库状态下不可直接扫码出库。

- 待出库：已被出库单拣选等待确认。转移至已出库通过确认出库或扫码出库。在库状态必须经过待出库才能到达已出库。

- 已出库：终态，不可再转移。已出库条码不可封存、解封、重新入库。

- FROZEN：封存暂停流转状态。仅可从在库状态进入。转移至在库通过解封操作。FROZEN状态下的条码不可被出库拣选，不可扫码入库。

## 6. 安全设计

### 6.1 认证机制

采用JWT无状态认证。用户登录成功后服务端签发包含用户ID、用户名、角色列表的令牌，使用HS384算法签名，有效期2小时。

前端将令牌存储于localStorage，每次请求通过Axios拦截器自动附加到Authorization头。

后端JwtInterceptor拦截所有非OPTIONS请求，校验令牌合法性。校验通过后将用户信息注入BaseContext线程变量。

### 6.2 密码安全

密码使用BCrypt算法加密存储，不可逆。登录时通过BCrypt.matches比对明文与密文。密码传输通过HTTPS保护。

### 6.3 API Key保护

AI服务的真实API Key存储在application-local.yml文件中。该文件已加入.gitignore排除，不会被提交到版本控制。application.yml中仅包含占位符demo-api-key。生产环境通过环境变量注入。

### 6.4 RBAC权限控制

系统采用基于角色的访问控制。用户关联角色，角色关联权限。权限粒度到功能模块级别。JWT令牌中携带角色列表，后端通过角色判断权限。
